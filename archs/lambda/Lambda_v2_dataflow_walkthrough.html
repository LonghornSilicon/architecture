Execution Scenario

The user has been chatting for a while. The KV cache is partially populated with prior-turn context. The chip is about to generate token #42. Model: Llama-3.2-3B — 3,072-dimensional hidden state, 28 transformer layers, 8 KV heads each of dimension 128.

Each numbered section below corresponds to one discrete stage of hardware activity, described in execution order. Where multiple blocks operate concurrently, the ordering reflects the logical dependency graph rather than strict serialization.

Stage 0
Boot Sequence and Weight DMA
HIF — Host Interface 0.30 mm²

When Lambda is plugged into a USB-C port, the first block to wake is the HIF (Host Interface) — a USB-C 2.0 endpoint running at 480 Mbps, occupying ~0.30 mm² at the die perimeter. HIF has three responsibilities at power-on: (1) enumerate as a USB device on the host bus; (2) expose a CSR (Configuration/Status Register) interface for driver-side register writes; and (3) provide a JTAG + scan-chain debug port.

The host driver performs two operations over HIF. First, it writes approximately 3,000 instructions of pre-compiled microcode into the LSU's instruction RAM — the static schedule for running Llama-3.2-3B. Second, it initiates a 1.5 GB DMA transfer of W4-quantized weights from host DRAM into the off-chip LPDDR5X package. At 60 MB/s sustained over USB-C 2.0, this transfer requires ~25 seconds but occurs only once per power-on cycle. Weights persist in LPDDR5X until the device is unplugged.

Doorbell Queue

Once armed, the chip awaits prompts via HIF's 16-deep doorbell queue. When the user submits a prompt, the host driver writes a doorbell entry; HIF asserts an interrupt; the LSU wakes and begins decode.

Stage 1
Layer Schedule Dispatch
LSU — Layer Sequencer Unit 0.10 mm²

The LSU (Layer Sequencer Unit) is the chip's control plane: a 3-stage in-order RISC pipeline (fetch → decode → dispatch) with 16 general-purpose registers and a 4 KB microcode RAM containing the entire pre-compiled execution schedule. Its minimalism is intentional — transformer decode is structurally identical across all 28 layers, so a static, deterministic program suffices. There is no branch predictor, no out-of-order execution, no cache hierarchy, and no virtual memory.

The first instruction issued for token #42 is:

LSU Microcode — Layer 0 Entry ISA-32 
LOAD_WEIGHTS  layer=0, slice=qkv_proj       ; DMA descriptor → MSC queue
ISSUE_MAT_E   qkv_proj, in=act_buf, out=qkv_scratch  ; dispatch MatE (stalls on weights)
ISSUE_VEC_U   rope, q, k                    ; pre-queued: runs after MatE
ISSUE_KCE_COMP k → kv_scratchpad            ; pre-queued: runs after VecU

LOAD_WEIGHTS is not a memory-load in the CPU sense — it writes a DMA descriptor into one of MSC's request queues and returns immediately. ISSUE_MAT_E hands off to the Matrix Engine, which stalls internally awaiting the incoming weight stream. The LSU continues speculatively, issuing subsequent instructions that queue behind the in-flight memory operation.

Design Principle — Static Schedule

The absence of runtime control flow is not a limitation — it is the architectural claim. A pre-compiled schedule with zero branches eliminates an entire class of verification complexity. The LSU is the conductor; from this point forward, all real computation occurs in the functional units.

Stage 2
Address Translation and DRAM Request Issuance
MSC — Memory Subsystem Controller 0.18 mm²

The MSC (Memory Subsystem Controller) receives the LSU's DMA descriptor and acts as the chip's memory traffic cop, binding together the SRAM crossbar, the LPDDR5X controller, the block-table TLB, and the DMA engine into a single arbitrated fabric.

Block-Table TLB

MSC first consults its 128-entry associative block-table TLB, which maps logical addresses of the form (session_id, layer_id, block_id) to physical SRAM bank offsets or DRAM row/column addresses. For layer-0 QKV weights, the mapping is static and pre-populated by the host driver at boot — a single-cycle TLB hit to a known DRAM address baked into the compiled schedule.

DRAM Command Sequencing

With the physical address resolved, MSC issues a DRAM read via the LPDDR5X controller. The controller serializes the request as a compliant LPDDR5X command sequence, honoring all timing parameters:

LPDDR5X Access Protocol — Row Hit Path
ACT  (bank_id, row_addr)  →  wait t_RCD  ≈ 14 ns
READ (col_addr)            →  wait t_CCD  ≈  5 ns
READ (col_addr + stride)   →  wait t_CCD  ≈  5 ns
  …                           (burst continues)
Data arrives at MSC read buffer  ≈ 30 ns after first READ

The controller uses an open-page policy: rows activated for one access are held open for subsequent column reads from the same row, avoiding redundant ACT commands. Bank-conflict avoidance logic tracks which banks are open across concurrent requests. The 4-port SRAM crossbar remains idle for this initial fetch — all compute units are stalled pending first-weight arrival.

Stage 3
Mixed-Signal Bus Driving
LPDDR5X × 16 PHY — Vendor IP 1.20 mm² (±0.30 mm² NDA-gated)

The LPDDR5X × 16 PHY is the only mixed-signal block on the die — licensed vendor IP from Synopsys DesignWare or Cadence Denali. At ~1.20 mm² it is the largest single block and the primary area risk item (R-Lv2-03). It sits at the die edge and exposes 16 data pads (DQ0–DQ15), 2 differential strobe pads (DQS/DQS#), differential clock pads (CK/CK#), and a command bus (CA0–CA6, CKE, CS).

When MSC issues an ACT command, the PHY's digital command path serializes it to the correct pin sequence and drives it across impedance-matched PCB traces to the off-chip LPDDR5X package. After tRCD (~14 ns), the row is open. On the READ command, the PHY's data receive path becomes active: its read-leveling logic uses the DQS strobes returning from the DRAM to time-align the 8,533 Mbps DQ bitstreams into the chip's 1 GHz synchronous clock domain via a continuous timing-recovery loop that compensates for temperature and voltage drift across the lifetime of the part.

Performance Bottleneck

8,533 Mbps × 16 lanes ÷ 8 bits/byte = 17 GB/s peak. After bank-conflict overhead and DRAM refresh, sustained throughput is ~12 GB/s. Every gate of compute on the chip is fed through this single memory channel. Bandwidth — not arithmetic capacity — is the end-to-end performance constraint at every operating point.

Risk Item R-Lv2-03 — PHY Area Uncertainty

The PHY footprint is quoted at 1.20 mm² with a ±0.30 mm² real range (NDA-gated; exact value requires written quote from Synopsys/Cadence at N16FFC). At 1.0 mm² (best case): SRAM budget grows to 1.0 MB with margin. At 1.5 mm² (worst case): total die exceeds 4 mm² — requires downgrading to LPDDR4x × 16 (losing the 4–5B model class) or growing the die to 4.5 mm² (~12% shuttle cost increase). Written quote required before floorplan freeze.

Stage 4
Weight Staging and SRAM Bank Overview
SRAM — 4-Bank On-Chip Memory 0.71 mm² total · 0.8 MB total

MSC writes the fetched weights into the weight_stream_buffer — a 50 KB single-port SRAM macro at the perimeter of MatE. This bank is deliberately undersized: at 12 GB/s with ~100 ns first-byte latency, the minimum buffer depth needed to hide DRAM latency is 1.2 KB. At 50 KB, the buffer provides 40× that minimum, enabling double-buffering with headroom to warm-cache hot weight slices (e.g., the LM head, which is accessed on every token).

Table 2. On-chip SRAM bank summary
Bank Name Capacity Ports Area (mm²) Contents / Role

activation_buffer 0.3 MB 2-port 0.26 Layer input/output; MatE reads while VecU writes next iteration
kv_scratchpad 0.4 MB 1-port 0.35 Headline bank. Compressed K and V for live context (~3,700 tokens/layer for Qwen2.5-3B at 3-bit)
codebook_const_rom 64 KB 1-port RO 0.05 Lloyd-Max centroids; RoPE frequency table; exp / rsqrt / sigmoid LUTs
Total 0.8 MB 0.71
KV-Dominant Allocation

The KV scratchpad holds 50% of total SRAM capacity. This is an explicit architectural decision: long-context decode performance is KV-bandwidth-bound, and the weight buffer needs only enough depth to hide DRAM latency — not to match the KV working set. This asymmetry is the primary differentiator from classical weight-centric accelerators.

Stage 5
QKV Projection — Weight-Stationary Systolic
MatE — Matrix Engine 0.10 mm² · 0.13 TOPS @ 1 GHz

The MatE (Matrix Engine) is an 8×8 array of Processing Elements (PEs), each containing one INT8×INT4 multiplier, one INT16 accumulator, and the latches required for weight-stationary dataflow. The array occupies 0.10 mm² of die — deliberately compact, because on a bandwidth-bound chip, compute area buys diminishing throughput.

PE Array Microarchitecture — Weight-Stationary Mode
       a₀   a₁   a₂  ···  a₇      ← activations stream right (INT8, new row each cycle)
        ↓    ↓    ↓        ↓
W₀₀ → [PE]─[PE]─[PE]─ ··· ─[PE]
        ↓    ↓    ↓        ↓
W₁₀ → [PE]─[PE]─[PE]─ ··· ─[PE]     Each PE: acc += weight × activation
        ↓    ↓    ↓        ↓         weight pinned for entire tile duration
       ···  ···  ···       ···
        ↓    ↓    ↓        ↓
W₇₀ → [PE]─[PE]─[PE]─ ··· ─[PE]
        ↓    ↓    ↓        ↓
       o₀   o₁   o₂  ···  o₇      ← INT16 partial sums drain from bottom edge

For the QKV projection of Llama-3.2-3B (3,072-dim hidden → 4,096-dim QKV combined), the schedule tiles the 3,072×4,096 GEMM into 8×8 chunks and streams them through the array. On each tile: 64 INT4 weight values latch into the 64 PEs (one per PE — this is weight-stationary). Then activations stream in from the left edge in rows of 8 INT8 values per cycle, staggered by one cycle per row. Every PE multiplies its pinned weight by its arriving activation, accumulates the result into its INT16 register, and passes the activation rightward to the next PE in the row. Partial sums drain from the bottom edge of the array.

Throughput Analysis

64 PEs × 2 operations/cycle (multiply + accumulate) × 1 GHz clock = 128 GOPS peak. The 3,072-dim QKV projection on a single token requires ~25M multiply-accumulate operations, completing in ~0.2 ms. Compute headroom over the bandwidth-bound floor is 1.6× constant — MatE idles waiting for the next weight batch at every operating point. Bandwidth, not arithmetic capacity, governs end-to-end throughput.

Output: qkv_scratch — three 3,072-element vectors (Q, K, V) written to activation_buffer.

Stage 6
Rotary Positional Embedding (RoPE)
VecU — Vector Unit 0.144 mm² · 8-lane FP16/BF16 SIMD

LSU dispatches ISSUE_VEC_U rope, q, k. The VecU (Vector Unit) is the chip's programmable SIMD engine: 8 lanes operating on FP16 or BF16 scalars, sharing a 1 KB microcode RAM and 4 KB of transcendental lookup tables (exp, rsqrt, sigmoid — 64 entries each with linear interpolation logic). Every non-GEMM operation on the chip is implemented as a microcoded loop in VecU.

RoPE encodes token position into the query and key vectors by rotating consecutive element pairs by a frequency-dependent angle. For a 64-dimensional head, the head is partitioned into 32 pairs (x, y), and each pair undergoes the rotation:

(x, y)  →  (x·cos θk − y·sin θk ,   x·sin θk + y·cos θk)

Angles θk are read from the RoPE frequency table in codebook_const_rom. VecU's microcode sequence for RoPE is ~12 µops: load pair → load cos/sin from ROM → 4 multiplies + 2 adds → store. Eight lanes process 8 pairs in parallel, completing the 32 pairs per head in 4 microcode iterations. Both Q and K are rotated, for a total of ~30 µops per token's RoPE application.

Design Note — Programmable over Fixed-Function

VecU's programmability eliminates four separate fixed-function blocks (RoPE, softmax, RMSNorm, SiLU) in favor of one verifiable SIMD engine. The microcode footprint — 1 KB — is negligible. The verification surface reduction is substantial. This was a deliberate area/complexity trade-off in the v2 architecture.

Stage 7
KV Compression via TurboQuant
KCE — KV Compression Engine 0.08 mm² · zero multipliers

LSU fires ISSUE_KCE_COMP k → kv_scratchpad. The KCE (KV Compression Engine) is Lambda v2's primary IP contribution — the silicon implementation of TurboQuant. At 0.08 mm², it achieves 3.0× KV compression with no multiplication hardware whatsoever.

K is a 1,024-element FP16 vector (8 KV heads × 128 dim) for this token in this layer. KCE processes it in 16-element chunks. The pipeline for each chunk:

KCE Pipeline — Per 16-Element Chunk
┌─────────────────────────────────────────────────────────────────────────┐
│  [16 × FP16 input elements]                                             │
│                    ↓                                                    │
│  16-pt Walsh-Hadamard Transform                                         │
│    4 butterfly stages × 8 add/sub pairs = 32 adds/subtracts            │
│    Effect: spreads outlier energy uniformly across all 16 coordinates   │
│                    ↓                                                    │
│  Nearest-Centroid Classifier                                            │
│    7 comparators × 16 lanes = 112 comparators in parallel              │
│    Selects closest of 8 codebook entries per coordinate → 3-bit index  │
│                    ↓                                                    │
│  Bit-Pack                                                               │
│    16 × 3-bit indices = 48 bits + 16-bit magnitude header = 64 bits    │
│    Compression: FP16 (256 bits) → 64 bits = 4.0× raw; ~3.0× effective │
│                    ↓                                                    │
│  [Write 8 bytes → kv_scratchpad]                                        │
└─────────────────────────────────────────────────────────────────────────┘

Walsh-Hadamard Transform

The Hadamard butterfly is a specific arrangement of additions and subtractions that is mathematically equivalent to multiplying the input by an orthogonal rotation matrix with ±1 entries. This rotation "spreads" outlier energy: if the input vector has one dominant coordinate (a common pathology in transformer KV caches), the butterfly redistributes that energy roughly uniformly across all 16 coordinates, producing a distribution that is approximately Beta-distributed — the target regime for the Lloyd-Max codebook.

Lloyd-Max Codebook

The codebook holds 8 centroids (64 bytes stored in codebook_const_rom), computed offline to be optimal for the Beta distribution. Each post-Hadamard coordinate independently selects the nearest centroid using a bank of 7 comparators. The resulting 3-bit index, combined with a 16-bit group-wise magnitude header, achieves ~3 bits per element effective compression versus FP16's 16 bits per element — a 3.0× storage and bandwidth reduction. The inverse path (3-bit index → FP16 approximate) is a ROM lookup used during attention scoring (§8).

Zero-Multiplier Architecture

Every operation in the KCE pipeline is an add, subtract, comparator evaluation, or ROM lookup. There are no multipliers. This is why the entire block fits in 0.08 mm² and consumes only ~50 mW at full throughput. The compressed K, and then V, are written to kv_scratchpad alongside all prior-token KV vectors for this layer.

Stage 8
Compressed-Domain Q·KT Scoring
MatE — output-stationary mode INT8 Q × INT3 K

LSU dispatches ISSUE_MAT_E qk_dot, q, k_compressed → scores. MatE switches from weight-stationary to output-stationary mode by asserting a single CSR mode bit, which changes which tensor is pinned in the PE array and which streams through.

In output-stationary mode, Q is pinned (8 query heads, one per row of the array) and the compressed past-K vectors from kv_scratchpad stream through column by column. For a context of ~3,000 accumulated tokens, this streams 3,000 compressed-K vectors through the array, producing a scores tensor of shape [3,000 × 8 query heads].

Compressed-Domain Dot Product

For a symmetric quantization scheme, the dot product ⟨Q, dequant(Kcompressed)⟩ is algebraically equivalent to dequant_scale × ⟨Q, Kcompressed⟩. The scale is a scalar applied once per row sum — not per multiply. This means MatE executes the full attention scoring in the compressed domain: INT8 Q × INT3 K multiplications with INT16 partial sum accumulation. No FP16 dequantization step is required before scoring, and no intermediate FP16 K matrix is materialized.

Stage 9
Online FlashAttention-3 Softmax
VecU — online algorithm ~3,200 µops per layer at 3K context

LSU fires ISSUE_VEC_U softmax_online, scores. This is the most algorithmically significant microcode on the chip.

Naïve softmax requires materializing the full attention score matrix prior to computing the denominator: exp(scores − max(scores)) / Σ exp(scores). At 3,000 tokens × 8 heads × 2 bytes (FP16), that matrix occupies ~50 KB — saturating the activation buffer. The FlashAttention-3 online algorithm avoids full materialization by processing scores in tiles and maintaining only two running scalars per attention row:

FlashAttention-3 Online Recurrence — Per Query Row
State maintained per row:  (m_i, l_i, O_i)
    m_i  — running maximum of all scores seen so far
    l_i  — running sum of exp(score − m_i) for all scores seen
    O_i  — running weighted value accumulator

On each new tile of scores (s₁, s₂, ..., s_T):
    m_new  = max(m_i,  max(s₁...s_T))
    l_i   *= exp(m_i − m_new)                 ← rescale for new max
    O_i   *= exp(m_i − m_new)                 ← rescale accumulator
    m_i    = m_new
    l_i   += Σ exp(sⱼ − m_new)
    O_i   += Σ exp(sⱼ − m_new) · V_tile[j]   ← accumulate V contribution

After final tile:  output = O_i / l_i        (one scalar division per row)

The rescaling factor exp(m_old − m_new) corrects for having used the wrong maximum in all prior tiles — the algebra is exactly equivalent to full softmax. Intermediate storage per attention row is only (m_i, l_i): two scalars. No score tensor is ever fully materialized.

VecU's 8 lanes process 8 query-row states in parallel. The microcode loop is ~32 µops per tile: load tile → max-reduce → exp via LUT → multiply-accumulate → update state. Over 3,000-token context with tile size 32, approximately 100 tiles run per layer, for ~3,200 µops per softmax invocation. The exp() lookup is a 64-entry ROM spanning x ∈ [−16, 0] with linear interpolation to 0.05 ULP error — hardware cost is one 64×16-bit ROM and ~50 gates of interpolation logic.

Stage 10
Tiled Softmax × V Accumulation
MatE + VecU — pipelined

In the FlashAttention-3 tiled algorithm, the softmax_scores · V accumulation is not a separate GEMM invocation — it is interleaved with the softmax tile loop within each iteration of Stage 9. On every tile:

  1. MatE scores Q·KT for the current tile of past-K vectors from kv_scratchpad.
  2. VecU updates the running state (mi, li) and rescales Oi.
  3. VecU accumulates O_i += softmax_weight_j · V_tile[j] for each tile position.

MatE and VecU run concurrently, with VecU consuming the score tiles that MatE produces. The LSU's pre-compiled schedule explicitly orchestrates the cycle-by-cycle handoff between the two units. When the final context tile is consumed, O_i / l_i yields the exact attention output with no intermediate score tensor ever resident in SRAM.

After the last attention tile, attn_out resides in activation_buffer. The LSU then immediately dispatches ISSUE_MAT_E attn_out_proj — a 1,024 × 3,072 GEMM projecting the multi-head attention output back to the residual stream dimension, using the same weight-stationary dataflow as Stage 5.

Stage 11
Feed-Forward Network — Up / Gate / Down
MatE + VecU — sequential ~70% of layer compute · ~70% of layer LPDDR traffic

Three dispatches in rapid succession constitute the feed-forward sublayer. Collectively they represent the dominant compute and bandwidth cost of every transformer layer.

Table 3. FFN sublayer dispatch sequence — Llama-3.2-3B
#DispatchOperationShapeNotes
1 ISSUE_MAT_E ffn_up MatE: hidden → intermediate 3,072 × 8,192 ~2.67× expansion; largest GEMM in the layer
2 ISSUE_VEC_U silu_mul VecU: SiLU activation gate 8,192 elements SiLU(x) = x · σ(x); sigmoid LUT from codebook_const_rom
3 ISSUE_MAT_E ffn_down MatE: intermediate → hidden 8,192 × 3,072 Second-largest GEMM; weights stream from LPDDR

Each MatE dispatch pulls a fresh weight slice from LPDDR5X via the weight_stream_buffer at full 12 GB/s throughput. MatE runs at 50% utilization on average (weight-bound). VecU's SiLU pass processes 8,192 elements across 8 lanes in ~50 µops, with the sigmoid computed via the 64-entry LUT in codebook_const_rom. The remaining 30% of layer time — QKV projection, attention output projection, normalization — uses the same hardware in the same order.

Stage 12
RMSNorm and Residual Addition
VecU — ~40 µops

LSU dispatches ISSUE_VEC_U rmsnorm_residual. VecU reads the current layer's residual stream from activation_buffer, adds the FFN output, computes the RMS over the 3,072 elements, looks up 1/√RMS from the rsqrt LUT, multiplies through element-wise, and writes the result back. The microcoded sequence is ~40 µops; the reciprocal-square-root lookup is the hot transcendental operation on this path.

Layer Complete

activation_buffer now holds the normalized, residual-summed output of layer 0 — the input vector for layer 1. The LSU's program counter increments to the layer-1 entry point. Llama-3.2-3B has 28 layers; each costs ~5 ms of LPDDR-bound time on this chip. Per-token latency: 28 × ~5 ms ≈ 134 ms/token → ~7.4 tok/s. Execution loops back to Stage 1 with layer = 1.

Stage 13
LM Head, Sampling, and Token Emission
MatE · VecU · HIF

After layer 27, LSU fires ISSUE_MAT_E lm_head — a 3,072 × 128,256 projection from the final hidden state to the full vocabulary logit space. This is MatE's largest single GEMM by output dimension, though the input is still a single token vector (3,072 elements), so the weight-streaming overhead dominates compute. Weights for the LM head benefit from the warm-cache behavior of the weight_stream_buffer, which retains frequently-used hot slices across tokens.

VecU runs sampling over the 128,256-element logit vector (~30 µops): temperature scaling, top-k masking, top-p nucleus truncation, and a single softmax + categorical draw. The chosen token ID — a 17-bit integer — is placed in HIF's transmit buffer.

HIF packetizes the token as a USB-C 2.0 bulk transfer and delivers it to the host driver over the cable. The runtime (llama.cpp, a custom Python loop, or equivalent) receives the token ID and renders it on screen.

The LSU immediately resets its program counter to layer 0, seeds activation_buffer with the new token's embedding, and begins the next decode pass. The token-to-token loop runs ~125–135 ms per token — 7–8 tok/s sustained.
