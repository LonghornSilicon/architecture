Lambda v3 — All-SRAM Tiny-LLM Accelerator

No off-chip memory. No PHY. No mixed-signal IP. The entire model lives on-die.
Process: TSMC N16FFC  ·  Die: 2.0 × 2.0 mm = 4 mm²  ·  Off-chip memory: NONE
Target workload: 2-10M-parameter transformers (TinyStories-class, distilled BERTs, voice-intent classifiers)
Power envelope: 0.25 W always-on / 0.5 W interactive / 1.5 W peak
Shuttle: IMEC / Europractice mini@sic 2.0 (primary) — Muse Semiconductor (US fallback)
Companions: Lambda_v3_4mm2.yaml  ·  Lambda_v2_4mm2.yaml (3-5B alternative)  ·  Lambda_v1_4mm2.yaml (coprocessor alternative)

Drawn at 250 pixels = 1 mm. Die is 2.0 × 2.0 mm = 4.0 mm² with a 100 µm I/O ring. v3's defining feature is the absence of an LPDDR PHY — that 1.2 mm² of die that v2 spends on talking to off-chip DRAM is, in v3, repurposed for on-die SRAM. The entire model lives in the chip.

The architectural bet: at 4 mm² with no PHY, we can fit ~2.5 MB of SRAM. At TurboQuant 3-bit weights, that's up to 6.7M parameters entirely on-die. At W2 with rotation, ~10M. This is small for a "real LLM" — but it's exactly the sweet spot for always-on edge AI: wake-word detection, voice-intent classification, TinyStories-class generative writing, distilled classifiers. The chip runs in a 0.25 W envelope (sub-watt), enabling fanless / battery / coin-cell deployment that no LPDDR-equipped chip can match.
How v3 differs from v1 and v2:
  • v1 = KV/attention coprocessor; pairs with a host CPU running the model. No LPDDR. Needs software stack to demo.
  • v2 = standalone with LPDDR5X x16 PHY. Runs 3-5B models. PHY is 30% of die area.
  • v3 (this chip) = standalone with NO off-chip memory. Runs 2-10M models. PHY-less; everything on-die.
All three are 4 mm². Same shuttle cost. Same TSMC N16FFC process. Different architectural philosophies.

The structural inversion vs v2: v2 spends 65% of die on infrastructure (PHY + I/O ring + clock + host) and only 35% on compute + memory. v3 spends 31% on infrastructure and 67% on compute + memory. Same die size, totally different chip.

3. What models actually run on v3

Compute-bound regime. tok/s = MatE_GOPS × util / (params × 2 ops/param). MatE 4×4 = 32 GOPS peak, ~19 GOPS at 60% util. SRAM bandwidth is essentially infinite (~256 GB/s on-die) — bandwidth never bottlenecks.

Largest reasonable model: ~8M params with W2-rotation quantization (TinyStories-8M class). At W3 (TurboQuant-style) the ceiling drops to ~5M. At W4 (standard INT4) the ceiling drops further to ~4M. The right model class for v3 is specialized 4-8M-parameter networks: voice intent, distilled classifiers, narrow-domain assistants, TinyStories-class generators.

The case for v3: lowest verification load, lowest cost, fastest to silicon, no mixed-signal IP integration, sub-watt power envelope, no software stack required for demo. The trade is model class — v3 only runs networks 100-1000× smaller than v2.

When v3 is right: sponsor or paper interest is in always-on edge AI / IoT / wearables / battery-powered LLM, OR the team needs the lowest-risk fastest-to-silicon Lambda variant for credibility before attempting v2. When v3 is wrong: demo story requires "real LLM" capability; customer wedge needs LPDDR-class memory; sponsors want H100-comparable workloads.
